# helpdora — agent guide

A small Node CLI that translates `<cmd> --help` / `man <cmd>` into Japanese, or answers natural-language questions about commands, by talking to LM Studio (or any OpenAI-compatible endpoint).

The package is published as [`helpdora`](https://www.npmjs.com/package/helpdora) on npm. The binary name is also `helpdora`. The local working directory is still called `dora/` (legacy) — don't be confused by that.

## Quick commands

```bash
pnpm dev <args>      # tsx src/cli.ts <args>  (development mode)
pnpm test            # vitest run             (must pass on Ubuntu CI)
pnpm typecheck       # tsc --noEmit
pnpm build           # tsup → dist/cli.js
```

## Layout

- `src/` — TypeScript source. Edit here.
- `src/tone/doraemon.ts` — Doraemon-style prompt fragments. The `--dora` flag and `--tone dora` value enable this tone; **these names refer to the character, not the package**, so don't rename them.
- `src/modes/` — one file per mode (summary / translate (full) / prompt (lookup+intent) / precache / install-skill / completion).
- `src/providers/cli-subprocess.ts` — claude/codex CLI bridge for `--provider claude|codex`.
- `dist/` — generated by `tsup`. Gitignored.
- `test/` — vitest tests. Runs on Ubuntu in CI.

## Testing notes

- Tests must run on Ubuntu (GitHub Actions). **Never hardcode Windows paths** like `C:\dev\src\dora` in `cwd:` or anywhere else — `spawnSync` should inherit the parent cwd. There was a real bug here (fixed in #1).
- `test/help-fetcher.test.ts` shells out to `node` for help output. That's portable; don't replace with platform-specific helpers.
- The CLI tests run the binary via `process.execPath --import tsx src/cli.ts`. Keep that pattern when adding new CLI tests.

## Configuration namespaces

These are part of the public API. Don't rename without bumping a major:

- Config dir: `~/.config/helpdora/config.json` (uses `XDG_CONFIG_HOME`)
- Cache dir: `~/.cache/helpdora/` (uses `XDG_CACHE_HOME`)
- Env vars: `HELPDORA_PROVIDER`, `HELPDORA_BASE_URL`, `HELPDORA_API_KEY`, `HELPDORA_MODEL`, `HELPDORA_TIMEOUT_MS`
- Skill name: `~/.claude/skills/helpdora/SKILL.md`
- zsh completion install path: `~/.zfunc/_helpdora`

## Release flow

We use [release-please](https://github.com/googleapis/release-please) on `main`. **Don't manually edit `package.json` version, don't manually `git tag`.**

### How releases happen

1. You merge a PR with a [Conventional Commits](https://www.conventionalcommits.org/) message (`feat:`, `fix:`, `feat!:`, etc.) into `main`.
2. `.github/workflows/release.yml` runs release-please-action.
3. release-please opens (or updates) a PR titled `chore(main): release helpdora X.Y.Z` with the version bump + `CHANGELOG.md` entries.
4. When you merge that PR, release-please tags `vX.Y.Z`, creates a GitHub Release, and the same workflow publishes to npm via OIDC (Trusted Publisher — no token, with provenance).

### Conventional commits cheat sheet

| Prefix       | Effect       | Example                                   |
| ------------ | ------------ | ----------------------------------------- |
| `feat:`      | minor bump   | `feat: zsh補完を改善`                     |
| `fix:`       | patch bump   | `fix: cache key collision`                |
| `feat!:` / `BREAKING CHANGE:` | major bump | `feat!: rename --provider option` |
| `chore:` / `docs:` / `refactor:` / `test:` / `ci:` | no bump | `chore: deps update` |

Use these on the **squash-merge title** (set the PR title to the conventional commit message — that's what release-please reads).

### Baseline

`release-please-config.json` pins `bootstrap-sha` to the `v0.2.0` baseline so prior history doesn't re-trigger releases. `.release-please-manifest.json` tracks the current released version. release-please updates the manifest itself; don't edit it by hand.

## Branch protection

- `main` requires PR (squash merge only, linear history)
- `test` job in `.github/workflows/ci.yml` must pass
- No force push, no deletion
- Admin (you) can bypass — but don't, except for plumbing emergencies

## Local quirks

- Windows users: line-ending warnings (`LF will be replaced by CRLF`) are noise and safe.
- `.envrc` is gitignored. It's used to hold an npm bootstrap token; keep it out of commits.
- `pnpm-lock.yaml` is committed. CI runs `pnpm install --frozen-lockfile` so any dependency change must update the lockfile.
